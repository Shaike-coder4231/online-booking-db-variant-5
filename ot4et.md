ОТЧЕТ ПО ПРАКТИЧЕСКОЙ РАБОТЕ
Тема: Онлайн-запись в салон красоты с выбором мастера и дополнительных услуг

========================================================================
1. ОПИСАНИЕ ПРОЕКТА И НАЗНАЧЕНИЕ
========================================================================
Разработана реляционная база данных для автоматизации онлайн-записи клиентов в салон красоты. Система позволяет выбирать конкретного мастера, добавлять основные и дополнительные услуги, фиксировать обязательные предварительные консультации и контролировать расходные материалы на складе. При создании записи реализована логика проверки наличия продуктов, необходимых для выбранных услуг.

========================================================================
2. СТРУКТУРА БАЗЫ ДАННЫХ
========================================================================
База данных online_beauty_salon_variant5 состоит из 10 таблиц:
- clients (клиенты)
- masters (мастера)
- services (основные услуги)
- appointments (записи на приём)
- appointment_services (связь записей с услугами)
- products (склад расходных материалов)
- service_products (нормы расхода продуктов для услуг)
- consultations (предварительные консультации)
- additional_services (дополнительные услуги)
- appointment_additional_services (связь записей с доп. услугами)

========================================================================
3. ОПИСАНИЕ ТАБЛИЦ
========================================================================
clients:
  client_id - первичный ключ, автоинкремент
  last_name, first_name, patronymic - ФИО клиента
  phone, email - уникальные контактные данные
  birth_date - дата рождения
  created_at - время регистрации

masters:
  master_id - первичный ключ, автоинкремент
  last_name, first_name - ФИО мастера
  specialization - перечень специализаций (парикмахер, косметолог, маникюр, массажист, стилист)
  phone - контактный телефон
  rating - рейтинг от 0 до 5
  is_active - статус занятости

services:
  service_id - первичный ключ, автоинкремент
  service_name - уникальное название услуги
  price - стоимость в рублях
  duration_minutes - длительность в минутах
  requires_consultation - флаг обязательной консультации
  description - текстовое описание

appointments:
  appointment_id - первичный ключ, автоинкремент
  client_id, master_id - внешние ключи на клиентов и мастеров
  appointment_datetime - дата и время приёма
  status - статус записи (запланировано, в_процессе, завершено, отменено)
  total_price - итоговая стоимость
  created_at - время создания записи

appointment_services:
  appointment_service_id - первичный ключ
  appointment_id, service_id - внешние ключи
  quantity - количество повторений услуги в записи
  price_at_booking - зафиксированная цена на момент бронирования

products:
  product_id - первичный ключ
  product_name - уникальное название материала
  unit - единица измерения
  stock_quantity - текущий остаток на складе
  min_stock_level - минимальный допустимый уровень
  price_per_unit - закупочная цена

service_products:
  service_product_id - первичный ключ

consultations:
  consultation_id - первичный ключ

additional_services:
  additional_service_id - первичный ключ
  service_name - название

appointment_additional_services:
  appointment_additional_id - первичный ключ

========================================================================
4. СВЯЗИ МЕЖДУ ТАБЛИЦАМИ
========================================================================
- Один клиент может иметь много записей и много консультаций (связи 1:M)
- Один мастер может принимать много клиентов и проводить много консультаций (1:M)
- Одна запись содержит одну или несколько основных услуг через таблицу-связку appointment_services (M:M через промежуточную таблицу)
- Одна запись содержит одну или несколько дополнительных услуг через appointment_additional_services (M:M через промежуточную таблицу)
- Каждая услуга требует определённого набора продуктов, что описывается в service_products (M:M между services и products)
- При удалении клиента его консультации удаляются каскадом, а записи блокируются для сохранения истории
- При удалении мастера или услуги операция блокируется, если есть активные записи

========================================================================
5. НОРМАЛИЗАЦИЯ
========================================================================
База данных приведена к третьей нормальной форме (3NF):
- 1NF: все поля содержат атомарные значения, нет массивов или повторяющихся групп
- 2NF: в таблицах с составными ключами (например, appointment_services) все атрибуты зависят от полного ключа, частичных зависимостей нет
- 3NF: исключены транзитивные зависимости. Например, цена услуги хранится только в таблице services, а не дублируется в записях. Нормы расхода вынесены в отдельную таблицу service_products

========================================================================
6. ОГРАНИЧЕНИЯ ЦЕЛОСТНОСТИ
========================================================================
- PRIMARY KEY во всех таблицах с автоинкрементом
- FOREIGN KEY с настройками ON DELETE CASCADE или ON DELETE RESTRICT в зависимости от бизнес-логики
- UNIQUE на полях phone и email клиента, на названии услуг и продуктов, на временных слотах мастера
- CHECK ограничения: возраст клиента не менее 14 лет, рейтинг мастера от 0 до 5, цена услуг и продуктов строго положительная, количество на складе не отрицательное, длительность услуги от 15 до 240 минут
- NOT NULL на всех обязательных полях ввода

7. ИНДЕКСЫ
========================================================================
Созданы следующие индексы для ускорения выборки:
- idx_appointments_datetime: поиск записей по дате и времени
- idx_appointments_client: быстрый доступ к истории клиента
- idx_appointments_master: формирование расписания мастера
- idx_services_consultation: фильтрация услуг, требующих консультации
- idx_products_stock: контроль складских остатков
- idx_appointment_services: оптимизация соединений при анализе состава записей

8. SQL-ЗАПРОСЫ И ИХ НАЗНАЧЕНИЕ
========================================================================
Запрос 1: Услуги, которые чаще всего записываются вместе
  Использует само-соединение таблицы appointment_services по полю appointment_id с условием as1.service_id < as2.service_id для исключения дублей. Группирует пары услуг и сортирует по частоте совместного выбора. Применяется для маркетинговых акций и рекомендаций клиентам.

Запрос 2: Проверка остатков продуктов при записи
  Соединяет записи, услуги, нормы расхода и склад. Сравнивает требуемое количество с доступным остатком. Выводит статус OK или NOT_ENOUGH. Позволяет администратору заранее закупать материалы или предупреждать клиента о задержках.

Запрос 3: Аналитика загрузки мастеров
  Агрегирует данные по каждому мастеру: количество приёмов, общая выручка, средний чек, число уникальных клиентов. Используется для расчёта премии, планирования графика и оценки эффективности специалистов.

9. ТЕСТИРОВАНИЕ ОГРАНИЧЕНИЙ
========================================================================
Проведена проверка корректности работы ограничений:
- Двойная запись на одно время к одному мастеру отклоняется системой
- Удаление мастера с активными записями блокируется
- Регистрация клиента младше 14 лет невозможна
- Попытка установить отрицательную цену услуги или рейтинг выше 5 отклоняется
- Дублирование email или телефона не допускается
- Каскадное удаление консультаций при удалении клиента работает корректно
- Проверка складских остатков корректно выявляет дефицит материалов
