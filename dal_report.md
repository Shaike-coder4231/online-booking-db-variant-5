1. АРХИТЕКТУРА УРОВНЯ ДОСТУПА К ДАННЫМ
========================================================================
Уровень доступа к данным реализован в виде набора PHP-классов, инкапсулирующих все взаимодействия с СУБД MySQL. Архитектура построена по принципу разделения ответственности и состоит из четырех основных компонентов:

1. Конфигурация (config.php): Содержит параметры подключения (хост, имя БД, пользователь, пароль, кодировка). Вынесена в отдельный файл для безопасности и удобства развертывания. Реальный файл игнорируется в Git, предоставляется шаблон config.example.php.

2. Менеджер соединений (Database): Реализует паттерн Singleton. Обеспечивает создание единственного экземпляра PDO-соединения на протяжении жизненного цикла скрипта. В конструкторе жестко задаются режимы безопасности: отключение эмуляции подготовленных выражений, выброс исключений при ошибках, установка режима выборки по умолчанию.

3. Абстрактный репозиторий (AbstractRepository): Базовый класс, предоставляющий универсальные CRUD-операции (findAll, findById, insert, update, delete). Содержит логику безопасной динамической сортировки через белый список разрешенных столбцов. Все конкретные репозитории наследуют этот класс.

4. Конкретные репозитории (ClientRepository, MasterRepository, ServiceRepository, AppointmentRepository): Реализуют специфическую бизнес-логику и сложные запросы, соответствующие предметной области салона красоты. Инкапсулируют SQL, работают только через подготовленные выражения PDO.

3. ВЫБОР ПАТТЕРНА: REPOSITORY vs DATA MAPPER
========================================================================
В проекте использован паттерн Repository. Выбор обусловлен следующими факторами:
- Оптимальное соотношение сложности и функциональности для учебных и средних проектов.
- Репозитории возвращают ассоциативные массивы, что упрощает сериализацию в JSON для API и снижает накладные расходы на создание объектов-сущностей.
- Бизнес-правила (проверка свободных слотов, учет консультаций) логично размещаются в методах репозитория, не перегружая слой представления.
- Паттерн Data Mapper потребовал бы создания отдельных Entity-классов с геттерами/сеттерами и маппинга полей, что избыточно для текущей задачи и усложнило бы архитектуру без практической выгоды.

4. ПЕРЕЧЕНЬ МЕТОДОВ РЕПОЗИТОРИЕВ И ИХ НАЗНАЧЕНИЕ
========================================================================
AbstractRepository:
- findAll(where, orderBy, orderDir, limit): Гибкая выборка с фильтрацией и безопасной сортировкой.
- findById(id): Получение записи по первичному ключу.
- insert(data): Вставка новой записи, возврат ID.
- update(id, data): Обновление полей по ID.
- delete(id): Удаление записи по ID.

ClientRepository:
- findByPhone(phone): Поиск клиента по уникальному номеру телефона.
- findByEmail(email): Поиск клиента по уникальному email.

MasterRepository:
- getActiveMasters(): Возвращает список активных мастеров, отсортированных по рейтингу.

ServiceRepository:
- getServicesRequiringConsultation(): Фильтрация услуг, требующих обязательной консультации.

AppointmentRepository:
- getAppointmentsByDate(date): Получение расписания записей на конкретную дату.
- getFreeSlots(masterId, date): Расчет свободных временных окон для мастера.
- createAppointment(...): Создание записи с проверкой занятости, расчетом стоимости и привязкой услуги (использует транзакции).
- updateStatus(id, status): Безопасное изменение статуса записи с валидацией ENUM-значений.

5. ЗАЩИТА ОТ SQL-ИНЪЕКЦИЙ
========================================================================
Безопасность работы с данными обеспечена на нескольких уровнях:
1. Подготовленные выражения PDO: Все пользовательские данные передаются исключительно через плейсхолдеры (:name). Конкатенация строк в SQL-запросах полностью исключена.
2. Отключение эмуляции: В настройках PDO установлено ATTR_EMULATE_PREPARES = false. Это гарантирует, что параметры отправляются на сервер MySQL отдельно от текста запроса, а не подставляются драйвером.
3. Белый список для ORDER BY: Динамическое имя столбца сортировки проходит проверку in_array() против жестко заданного массива разрешенных полей. Недопустимые значения автоматически заменяются на первичный ключ, что предотвращает инъекции через параметры сортировки.
4. Валидация бизнес-данных: Метод updateStatus() проверяет передаваемый статус по явному списку допустимых значений перед выполнением запроса.

========================================================================
6. РАБОТА С ТРАНЗАКЦИЯМИ
========================================================================
Транзакции реализованы в методе AppointmentRepository::createAppointment(), который затрагивает несколько таблиц. Алгоритм работы:
1. Вызов $pdo->beginTransaction() открывает транзакцию.
2. В блоке try выполняются последовательные операции:
   - Проверка уникальности временного слота (SELECT COUNT).
   - Получение стоимости услуги из таблицы services.
   - Вставка записи в таблицу appointments.
   - Вставка связи услуги в таблицу appointment_services.
3. При успешном завершении вызывается $pdo->commit(), фиксируя изменения.
4. При возникновении любого исключения (включая ошибки БД, дубликаты UNIQUE, сбои соединения) срабатывает catch, выполняется $pdo->rollBack(), отменяя все промежуточные изменения, и выбрасывается RepositoryException с понятным сообщением.
Это гарантирует атомарность операции: запись либо создается полностью со всеми связями, либо база данных возвращается в исходное состояние.

7. ВЫВОДЫ
========================================================================
В ходе работы успешно спроектирован и реализован уровень доступа к данным для системы онлайн-записи. Все SQL-запросы используют подготовленные выражения PDO, что полностью исключает риск SQL-инъекций. Динамические параметры запроса защищены белыми списками. Сложные операции инкапсулированы в транзакции, обеспечивающие целостность данных. Реализованный DAL готов к интеграции с веб-интерфейсом, REST API или MVC-фреймворком.
