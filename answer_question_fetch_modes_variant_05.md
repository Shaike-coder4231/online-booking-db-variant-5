# Ответ на контрольный вопрос: Режимы выборки PDO

**Выполнил:** Баженов Илья
**Группа:** 454
**Вариант:** 5
**Дата:** 20 мая 2026 г.

## Вопрос
Какие режимы выборки (fetch modes) предоставляет PDO? Какой режим вы выбрали по умолчанию и почему?

## Ответ

### 1. Доступные режимы выборки в PDO
PDO предоставляет константы для определения формата данных, возвращаемых методами `fetch()` и `fetchAll()`:

| Константа | Описание |
|-----------|----------|
| `PDO::FETCH_ASSOC` | Возвращает ассоциативный массив, где ключи — имена столбцов |
| `PDO::FETCH_NUM` | Возвращает нумерованный массив с индексами `0, 1, 2...` |
| `PDO::FETCH_BOTH` | Режим по умолчанию. Дублирует данные: доступны и по имени, и по индексу |
| `PDO::FETCH_OBJ` | Возвращает анонимный объект `stdClass` со свойствами, равными именам столбцов |
| `PDO::FETCH_LAZY` | Комбинация `FETCH_OBJ` и `FETCH_BOTH`. Свойства объекта создаются только при обращении |
| `PDO::FETCH_CLASS` | Создаёт экземпляр указанного класса и сопоставляет столбцы со свойствами |
| `PDO::FETCH_INTO` | Обновляет свойствами столбцов существующий экземпляр класса |
| `PDO::FETCH_BOUND` | Привязывает столбцы к PHP-переменным через `bindColumn()` |
| `PDO::FETCH_GROUP` / `FETCH_UNIQUE` / `FETCH_KEY_PAIR` | Модификаторы для группировки или индексации результатов |

### 2. Выбранный режим
В проекте установлен режим **`PDO::FETCH_ASSOC`** (настроен в `src/Database.php` через `PDO::ATTR_DEFAULT_FETCH_MODE`).

### 3. Обоснование выбора

1. **Читаемость и поддержка кода**
   Обращение `$row['client_id']` или `$row['appointment_datetime']` делает код самодокументируемым. В отличие от `FETCH_NUM`, где нужно помнить порядок столбцов в запросе, ассоциативные ключи исключают ошибки при рефакторинге SQL.

2. **Экономия памяти**
   `FETCH_BOTH` (режим по умолчанию в PDO) создаёт дублирующиеся данные в массиве, что увеличивает потребление памяти на ~30–40% при больших выборках. Для DAL, который может обрабатывать списки записей или товаров, `FETCH_ASSOC` оптимален по соотношению скорость/память.

3. **Архитектурное соответствие**
   Реализованный паттерн Repository возвращает данные в виде ассоциативных массивов. Это упрощает:
   - Сериализацию в JSON для REST API (`json_encode()` работает с массивами нативно)
   - Валидацию и трансформацию данных перед передачей в слой представления
   - Тестирование (массивы проще сравнивать в assert-проверках, чем объекты)

4. **Отказ от объектных режимов**
   `FETCH_OBJ` и `FETCH_CLASS` не использованы, так как в проекте не реализован паттерн Data Mapper/Active Record. Объекты `stdClass` не несут бизнес-логики, геттеров/сеттеров или валидации, поэтому их использование добавило бы избыточную абстракцию без практической пользы.

5. **Безопасность и предсказуемость**
   При использовании `SELECT *` порядок столбцов может измениться при миграциях БД. Ассоциативный доступ гарантирует, что код продолжит работать корректно без привязки к позициям.

### Вывод
`PDO::FETCH_ASSOC` выбран как наиболее сбалансированный режим для уровня доступа к данным: он обеспечивает высокую читаемость, экономию ресурсов, лёгкую интеграцию с API и полное соответствие архитектуре репозиториев, возвращающих структурированные массивы вместо сырых объектов или нумерованных индексов.
